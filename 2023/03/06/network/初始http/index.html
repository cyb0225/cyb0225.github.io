HTTP 状态码

大类:

  • 1xx 请求正处于处理阶段

  • 2xx 请求成功

    • 200 标准的请求成功
    • 204 请求成功但是没有body信息
    • 206 是应⽤于 HTTP 分块下载或断点续传，表示响应返回的 body 数据并不是资源的全部，而是其中的⼀部分。

  • 3xx 重定向，资源位置发生改变

    • 301 永久重定向

    • 302 临时重定向

      以上两种在返回响应头里有 Location 字段，指明后续要跳转的 URL.

    • 304 缓存控制，不需要跳转，客户端只需要使用本地缓存即可，后面有介绍 HTTP 缓存

    • 307 临时重定向，但请求方法不变

    • 308 永久重定向，但请求方法不变

  • 4xx 客户端发生的错误

    • 400 笼统的客户端错误
    • 403 客户端禁止访问该资源（可能是权限）
    • 404 请求资源不存在

  • 5xx 服务端发生的错误

    • 500 笼统的服务端错误
    • 501 功能还未支持
    • 502 网关代理错误
    • 503 服务繁忙，稍后重试

restful 协议

  • get 请求资源 （get 一般没有 body）
  • post 创建
  • put 更新
  • delete 删除

幂等：幂等（Idempotence）是指对同一个操作，若执行一次与执行多次的效果是相同的，就称该操作是幂等的。

换句话说，如果对于任意给定的输入，多次执行某个操作的结果都是一样的，那么该操作就是幂等的。简单来说，重复的操作结果不会发生变化或者不会产生二次影响的操作就是幂等的。

所以 put 和 get 应该是幂等的。

HTTP 缓存

  • 强制缓存

    • 通过 cache-control （相对时间）和 expires （绝对时间）来决定是否使用绝对缓存。
    • from memory cache
    • from disk cache

      cache-control 相对 expires 更灵活，所以 cache-control 在强制缓存优先级更高。

  • 协议缓存

    • If-Modified-Since + Last-Modified
      通过 Last-Modified 是否过期，过期返回 200 和新资源；未过期返回 304，客户端使用缓存资源。
    • If-None-Match + ETag
      通过客户端上传的 ETag 与服务端的 ETag 比较，判断资源是否被修改，被修改则返回 200 和新资源；反之返回304。

      ETag 对于 Last-Modified 来说更能保证客户端使用的缓存资源是最新的，且粒度更细（Last-Modified 的粒度为 1s）所以协议缓存 ETag 优先级更高

由于强制缓存是由浏览器来决定的，即浏览器判断 cache-control 或 expires 是否过期来决定是否使用缓存。所以默认先判断是否进行强制缓存，再请求服务器判断是否使用协议缓存。

浏览器调试技巧
ctrl + r 可以刷新浏览器，但是此时还是会走缓存的;如果不想走缓存，可以使用 ctrl + shift + r.
