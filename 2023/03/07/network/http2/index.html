HTTP 2 演进的技术

  1. 向前支持 HTTP1.1
    在 url 地址中 HTTP2 仍然使用 http 表示明文协议，https 表示加密协议。
    与 HTTP1.1 一致，协议的升级由服务端和客户端自己约定升级，便于 HTTP2 的推广。
    同时，只修改了应用层，底层还是基于 TCP 协议。体现网络分层架构的优势。

  2. 二进制协议

    为什么采用二进制编码代替 ASCII 编码？

    1. 相同信息容量下，二进制编码占用的 bit 位比 ASCII 更少。 例如，状态码 “200” 使用二进制表示是一个字节10001000，而用 ASCII 则需要三个字节。实际上，”Header: :status: 200 OK” 会被保存在头压缩的静态表里，所需空间更小。
    2. 原本 /r/n 、空格、冒号等用于分割的特殊字符也保存在静态表里，节省空间。
    3. 虽然 ASCII 更利于人类阅读，但是 HTTP2 基于 TLS 加密，传输过程中本身就已丢失可读性。

  3. 多路复用/并发传输

    同一个 TCP 连接，内部可以发送多个 HTTP 请求（并发）。解决了 HTTP1.1 中必须创建多个 TCP 请求才能并发处理请求的问题。同时，避免多次 TCP 握手与挥手。

    采用 二进制包裹原始数据，使不同的请求乱序在 TCP 传输仍能正确的被处理。

    细节有点多，我打算在阅读 go-http2 源码的时候写了，写完贴个 link 到这里。

  4. 优先级和依赖性

    针对多路复用的场景，服务端可以对流设置优先级，流的发送依据他的优先级排序。保证资源受限场景下核心服务正常工作，起到一定熔断的作用。

    客户端也可以设置 stream 流传输的依赖关系，并保存到一颗”优先级”树上，子流会在他的父流全部传输完全的情况下再去进行传输工作。

  5. 头压缩
    HTTP1.1 只支持了响应体 body 的压缩，但是头部占用字节仍然很大（如 cookies，User-Agent，Accpet），而且这部分数据大多数是固定重复的。所以 HTTP2 引入了头部压缩，并专门设计了 HPACK 压缩算法。HPACK 算法部分有点多，放到后面说。

  6. Reset

    针对需要有数据重发的场景，HTTP1.1 就需要断开重连 TCP 连接来实现。

    而 HTTP2 客户端支持发送 RST_STREAM 帧和新数据重置发错的数据。

  7. 服务端推送

    客户端请求资源X时，服务端很可能知道他下个要请求的资源是什么，会在发送完资源X后把相关的资源（可能是依赖关系）共同推送给客户端。客户端可以通过发送 RST_STREAM 帧来中断此次推送。

    HTTP2 的服务端推送能否代替 Websocket 的双工通信？
    答：显然是不能的。

  8. 流量控制

    HTTP2 建立在 TCP 上，为什么还要再实现流量控制？

仍然存在的问题

  1. 未完全解决线头阻塞的问题

    HTTP/2 是基于 TCP 协议来传输数据的，TCP 是字节流协议，TCP 层必须保证收到的字节 数据是完整且连续的，这样内核才会将缓冲区里的数据返回给 HTTP 应用，那么当「前 1 个 字节数据」没有到达时，后收到的字节数据只能存放在内核缓冲区里，只有等到这 1 个字节 数据到达时，HTTP/2 应用层才能从内核中拿到数据，这就是 HTTP/2 线头阻塞问题。

  2. HTTP2 基于 TLS 协议，每次创建连接需要进行 6 次握手。

  3. 一个 TCP 连接是由四元组(源 IP 地址，源端口，目标 IP 地址，目标端口)确定的，这意味着如果 IP 地址或者端口变动了，就会导致需要 TCP 与 TLS 重新握手，这不利于移动设备切换网络的场景，比如 4G 网络环境切换成 WIFI。

  • 动态字典

    存储动态数据

    由于常用的 Index 只有 61，且有些是 KV 组合，远远没有包含 HTTP Header 的全部，所以针对后面的部分，HTTP 设计了动态字典，将使用到的 Header 保存在 Index 为 62 及以后的位置中。

    例如，User-Agent 的 Value 大多是不一样的，就不能存在静态字典中。但是他又十分冗长，每次请求都是一样的。在发送完一次请求后，客户端和服务端就会对他进行一次哈夫曼编码，加入到动态字典中，存储仅需1 字节。

    所以，动态表是需要在同一个 TCP 连接中保存的，TCP 连接断开后就没了。而且如果连接一直不断开会导致动态表占用内存越来越大，此时会采用关闭 HTTP2 连接来释放内存。

    内存的阈值怎么判断？

    因为 Index 只有 8 位，所以能表示的值一共就 256 个，如果超过了这个值，HTTP2的 HeaderName:HeaderValue 又该如何安排？

    双方又是如何判断哪些字段需要放入动态表的，又是怎么做到统一的 Index？

  • 哈夫曼算法

    设计一棵树，使使用率越高的节点越靠近根的位置。按照先序遍历，左孩子标0，右孩子标1，对树的所有节点进行编码。这样越靠近根的节点所占字节数就越少。
